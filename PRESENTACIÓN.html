1️⃣ El Problema

  • ¿Cómo integran los LLMs herramientas externas?
  • APIs personalizadas = Cada servicio define su propio estándar
  • Sin estandarización = Código frágil, difícil de mantener
  • Imposible reutilizar herramientas entre clientes diferentes
Solución: Model Context Protocol (MCP)
Protocolo estándar JSON-RPC bidireccional que permite a un Client (IDE, LLM, etc.) interactuar de forma segura con un Server que expone datos y herramientas específicas.

2️⃣ ¿Qué es Model Context Protocol?

📋 Conceptos Clave:

  • Host: FastAPI + Dashboard
  • Client: MCPClient (solicita tools)
  • Server: MCPServer (expone tools)
  • Tools: Funciones ejecutables
  • Resources: Datos accesibles
  • Prompts: Plantillas reutilizables
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "generar_casos_prueba",
    "arguments": {...}
  }
}

JSON-RPC 2.0

3️⃣ Arquitectura de la Solución

Arquitectura Hexagonal (Puertos y Adaptadores)

Browser (Dashboard)
        ↓
   FastAPI Controller
        ↓
   Caso de Uso
        ↓
   Puerto del Dominio
        ↓
   Adaptador MCP
        ↓
   MCP Client
        ↓
   MCP Server
        ↓
   Tool MCP

Ventaja: Los casos de uso NO conocen MCP ni FastAPI. Solo dependen de puertos. Esto permite cambiar la implementación sin afectar la lógica.

4️⃣ Capas de Arquitectura

🌐 Web FastAPI, Controllers, Dashboard
⚙️ Application Casos de Uso, Orquestación
💼 Domain Entidades, Puertos (contratos)
🔧 Infrastructure MCP Server/Client, Adapters

⚡ Regla de Oro: Las dependencias apuntan hacia adentro. El Domain nunca conoce la Infrastructure.

6️⃣ Puertos (Contratos del Dominio)

  • TestingAnalysisPort: Genera casos de prueba
  • AcceptanceCriteriaPort: Genera criterios BDD (Given/When/Then)
  • RiskAnalysisPort: Analiza riesgos (funcionales, seguridad)
  • EffortEstimationPort: Estima esfuerzo QA
¿Por qué Puertos?
Permiten reemplazar la implementación (MCP, API, Base de Datos) sin cambiar los casos de uso. La inversión de dependencias es la clave de una arquitectura flexible.

7️⃣ Herramientas MCP (Tools)

Tool Input Output
generar_casos_prueba JiraTicket Test Cases (positivos, negativos, borde)
generar_criterios_aceptacion JiraTicket Given/When/Then structure
analizar_riesgos JiraTicket Riesgos funcionales, seguridad, integración
estimar_esfuerzo_qa JiraTicket Nivel (Alto, Medio, Bajo)

8️⃣ Flujo de Ejecución Paso a Paso

  1. Usuario llena formulario en Dashboard web
  2. JavaScript envía POST a `/testing` (o similar)
  3. FastAPI Controller recibe el request
  4. Controller crea entidad `JiraTicket`
  5. Controller llama al Caso de Uso correspondiente
  6. Caso de Uso llama al Puerto
  7. Adaptador MCP implementa el Puerto
  8. Adaptador usa `MCPClient.execute_tool()`
  9. MCPServer ejecuta la tool registrada
  10. Tool retorna resultado estructurado
  11. Resultado viaja de vuelta al Dashboard como JSON
  12. Usuario ve los artefactos QA generados

9️⃣ Endpoints API REST

POST /testing → Genera casos de prueba

POST /criteria → Genera criterios BDD

POST /risks → Analiza riesgos

POST /effort → Estima esfuerzo

Todos reciben:
{
  "titulo": "Agregar autenticación con Google",
  "descripcion": "Los usuarios deben poder iniciar sesión...",
  "tipo": "Story"
}

Responden con JSON estructurado (DTOs)

1️⃣5️⃣ Ventajas de MCP vs APIs Tradicionales

Aspecto APIs Tradicionales MCP
Estandarización Cada API define su estándar Protocolo unificado JSON-RPC
Tipado Estricto Depende del framework Esquemas JSON bien definidos
Seguridad Sin restricciones implícitas Límites claros definidos
Documentación Manual por cada API Auto-descubierta con list_tools

1️⃣8️⃣ Lecciones Aprendidas

✅ Qué Funcionó Bien

  • Arquitectura Hexagonal proporciona máxima flexibilidad
  • Uso de Puertos permite cambiar implementación fácilmente
  • MCP simplifica la integración LLM-herramientas
  • Separación de responsabilidades mejora testabilidad

⚡ Retos Técnicos

  • Manejo de procesos stdio requiere cuidado
  • Error handling en comunicación cliente-servidor es crítico
  • Testing de sistemas distribuidos es complejo

1️⃣9️⃣ Conclusiones

MCP es revolucionario porque:

  • 🎯 Estandarización: Un protocolo para todos
  • 🔒 Seguridad: Límites claros entre cliente-servidor
  • 📚 Reutilización: Tools usables por cualquier cliente
  • 🚀 Escalabilidad: Arquitectura desacoplada

El futuro de la integración LLM con herramientas pasa por MCP.
